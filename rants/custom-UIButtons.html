I like buttons, and switches, and levers, and knobs that turn or click (especially the ones that pop out). And don't get me started on pull-cords.

Why The System Button Works

To understand why .custom doesn't work when .system does, we need to take a quick look at the documentation (groan).

Hidden deep in the Creating buttons of a specific type subheading is the answer:

When creating a custom button — a button with the type UIButton.ButtonType.custom — the frame of the button is set to (0, 0, 0, 0) initially*. Before adding the button to your interface, you should update the frame to a more appropriate value.

There we have it. Both the X & Y positions are 0, which is fine. It's up there hanging out with the time. The problem is that both the width: & height: properties are also set to 0.

This is opposed to UIButton(type: .system) which has an intrinsic width and heigth based on the value of the .setTitle(_ title: String?, for: UIControl.State) as well some as default values set for other properties.

UIButtonConfiguration

The UIButton.Configuration will be the backbone for (almost) everything we need going forward. And all we need to get started are two lines.

I try to keep my init pretty short. One of the ways I like to do that is by abstracting out all of the properties that relate to appearance into a seperate method, which then gets called in the init.

On Line 16, I create a Method that contains an Instance of UIButton.Configuration.filled(). It will serve as the container for all of the properties we're about to mess around with.

.filled() is one of several static methods available within UIButton.Configuration. Each of them provide a set of default properties that are consistent with Apple's Human Interface Guidelines.

Back, Back, and Forth and Forth

Our love goes...

The default background looks bland. On top of that, gray tends to be a color associated with elements that are not user-interaction enabled. (Remeber the iPhone 4S wifi chip issues?)

Because I'm lazy, I'm going to use one of the UIColor options that UIKit gives us for free.

On the line following config.buttonSize = .medium, I'm going to add config.baseBackgroundColor = .link

There's an obvious problem you'll notice if the you run the project. The button's title is no longer visable. UIColor.link is the default tint / text color / title color for UIButtons.

Let's fix that.

Just below the line where we set the value of config.baseBackgroundColor, we're going to add the following:

config.baseForegroundColor = .systemBackground

Decisions, decisions...

We are constantly making decisions. What to eat, what to wear, who we spend time around.

As developers, we have to make decisions about the problems that we are trying to solve.

I'm probably telling on myself here, but I'm lazy. If there's an easy way to do something, I'm doing that. If it's already been done, I'll steal that code, install that package or SDK, etc. There's no point trying to reinvent something that's proven and effective.

Even doing all of this is a form of laziness. Imagine having to style each and every button in every view. My retort is simply: Nope. Not going to do it.

Take the time to understand what the problem is, consider your options, and take the most efficent path to resolution.

With all of that said, if you want to update the var config = UIButton.Configuration.bordered() to .filled(), by all means, go for it. I'm going to remain stubborn as an old goat and leave it as is.

Iconography

Often times, buttons will have an image that will provide additional context to the button's purpose. These can either be images from your Assets folder or from SF Symbols.

Again, lazy. SF Symbols.

Following the background and foreground colors, add this line: config.image = UIImage(systemName: "key.horizontal.fill") or whatever you'd like.

You should end up with something like this:

The image is there, but it feels a little cramped.

We can give the image a little breathing room by adding config.imagePadding = 6 to the line below the one where we set the image.

That's better, but what if we don't like where it's located? What if we wanted it to the right of the next? Above? Below?

config.imagePlacement is what we need. Here are your options:

config.imagePlacement = .top

config.imagePlacement = .leading

config.imagePlacement = .trailing

config.imagePlacement = .bottom
